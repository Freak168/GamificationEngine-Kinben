Tutorial for Kinben

This tutorial will show the first steps with the gamification engine Kinben. It describes how to create goals and their components which can be fulfilled in order to obtain rewards as well as the necessary prearrangements. Furthermore it explains how such a task can be offered to other players through a marketplace. Subsequently some other possible actions are presented.

All steps will be presented by taking the example of a developer named John Doe, who works for the organisation ”Test Organisation”. If some of the components already exist, you can continue with the next section.

In the following we will not state the full URI. We assume you will prefix all your requests with appropriate scheme, typically http, the domain, eventually a port and also your context path.

Eg.: http://localhost:8080/gamification_engine

1. Set up your account and organisation profile

Account

At the beginning you need to create an account. The email address and password are mandatory. The email address must not be already used in your database. Optionally you can also specify the first and last name.

Example: Creating an Account for John Doe with the e-mail address test@example.com and the password 123456.

POST

/account/?email=test@example.com&password=123456&firstName=John&lastName=Doe

Parameter

Purpose

email

A unique e-mail address which is connected with the account.

password

The password to validate the identity.

[firstName]

Optionally can be given a first name for the account.

[lastName]

Optionally can be given a last name for the account.

Response:

{

"content":{

"email": "test@example.com",

"password": "123456",

"firstName": "John",

"lastName": "Doe”

},

"contentResponseType": "OK",

"        info":[]

}

The parameters “firstName” and “lastName” can be changed with the following query:  

PUT

/account/?email=test@example.com&password=123456&firstName=Max&lastName=Power

Organisation

An organisation represents a specific company or a group of people participating in the gamification process. For each company, an API key is generated which is needed for all further interactions.

All database entries are associated with one unique organisation. For this purpose all further interactions are associated with an unique api key, which belongs to one organisation.

Eg: Creating an organisation, named “Test Organisation”. Email and password are required for authorisation.

POST

/organisation/?name=Test+Organisation&email=test@example.com&password=123456

Parameter

Purpose

name

The name of the organisation.

email

A unique e-mail address which is connected with the account.

password

The password to validate the identity.

Response:

{

"content":{

"        id": 1,

"managers":[

{"email": "test@example.com", "password": "123456", "firstName": "Max", "lastName": "Test"…}

],

"name": "Test Organisation",

"apiKey": "24bc004f-08ef-413d-988f-b2ca8cc39936"

},

"contentResponseType": "OK",

"        info":[]

}

As you can see, the authorized user is also the first manager of associated organisation.

In the following sections let’s assume that the API Key “24bc004f-08ef-413d-988f-b2ca8cc39936” was returned.

2. How to create a task

A task consists of several components which have to be created before the task itself. In the following this part will describe all the components and their order in which they have to be created.

In the following steps, the account, the organisation and its associated API key are those, which were created in the first one. If some of the components already exist, you can continue with the next section.

Role

A role describes which members of your organisation will be allowed to fulfil a particular task and get its rewards. Each Player can have many different roles such as a role for his occupation or the department in which he works.

Eg: Creating two roles: “manager” and “administration secretary” as an example for a specific group of employees. Another possibility would be to create one role for each department.

POST

/role/?roleName=manager&apiKey=24bc004f-08ef-413d-988f-b2ca8cc39936

/role/?roleName=administration secretary&apiKey=24bc004f-08ef-413d-988f-b2ca8cc39936

Parameter

Purpose

roleName

The name of a particular role. This can be the name of a department or a particular function.

apiKey

The API key which was generated for the organisation.

Player

A player represents an employee of your organisation and a user in the gamification application. By the creation each  player is assigned a nickname and certain roles. For each player a list for his rewards, finishedGoals and finishedTasks. The initial value of possible points, coins and levelindex is set to “0”. Furthermore an avatar can be used, by specifying the path of an image previously uploaded to a server. The format of such an image have to be .jpg or .png.

Eg: Creating a player named “Sunny” with the role “1” (manager). In the next line the previously returned id of this player is needed to create an avatar for him.

POST

/player/?nickname=Sunny&password=3456&reference=John Smith&roleIds=1&apiKey=24bc004f-08ef-413d-988f-b2ca8cc39936

PUT

/player/1/attributes/?id=1&attribute=avatar&value=https://www.testpath.jpg&apiKey=24bc004f-08ef-413d-988f-b2ca8cc39936

PlayerGroup

Players can be assigned to a Group. In dependent of your organization a group can be a department, a work group or several employees with the same occupation. It is possible to create special tasks which can be done also or only as a group. When a member of a group completed such a task the group obtains its rewards. Like a player, a group can be assigned an image as a logo. This can either be done when creating the group or later through a PUT query.

Eg: Creating a player group named “Group1” with two players as members and no path for an image.

Rewards

A Reward will be awarded in dependent of the goal for a group or one person after a person fulfilled a task. You can choose one or more of the following rewards for a task: A permanent reward like a badge or achievement which can be obtained only once. The other possibility is to award coins, points or a particular level.

Eg: Creating a rewards with 1000 points and in the second line a badge.

POST

/reward/?type=Points&name=Many many points&amount=1000&description=1000 Points for a Reward&apiKey=24bc004f-08ef-413d-988f-b2ca8cc39936

/reward/?type=Badge&name=Great Badge&icon=http://examplePath.png&description=A great badge for wisdom&apiKey=24bc004f-08ef-413d-988f-b2ca8cc39936

All possible types for a reward can be queried with:

GET

/reward/types?apiKey=24bc004f-08ef-413d-988f-b2ca8cc39936

The name of the reward.

amount

The amount of points or coins. When a badge or an achievement is created this parameter isn’t used.

icon

The url path for an image. This parameter is used for a badge or an achievement.

description

The description for the reward.

apiKey

The API key which was generated for the organisation.

Task

The description of a task contains the concrete steps which a player has to do to complete it. Depending on the rule of the goal some additional tasks may also have to be completed to fulfill the goal. A task can be used to create an offer in the marketplace, so that another player can do it and get the reward of it.

Goalrule

With a Goalrule can be defined which tasks and if a particular combination or only one task have to be fulfilled to reach a goal and obtain the rewards for it. When a rule is created such a combination in terms of a logical expression can be given. There are two types of rules that can be defined: a TaskRule or a PointsRule.

TaskRule: A TaskRule defines the combination of tasks. So for a defined sample of tasks either all of them have to be fulfilled (type: DoAllTaskRule) or only one (DoAnyTasksRule), for example only two tasks.

PointsRule: A PointsRule specify to reach a certain amount of points.

Eg: Creating a taskrule and in the second line a pointsrule.

POST

/rule/task?type=DoAllTasksRule&name=Do All&description=Do all tasks!&tasks=1,3&apiKey=24bc004f-08ef-413d-988f-b2ca8cc39936

/rule/point?name=200 Points&description=Reach 200 points&points=200&apiKey=24bc004f-08ef-413d-988f-b2ca8cc39936

Parameter

Purpose

type

Represents the type of a task rule. This can be DoAllTaskRule or DoAnyTasksRule. To fulfill DoAllTaskRule every task has to be completed, for DoAnyTasksRule only one task.

name

The name of the rule.

description

A detailed description for the rule.

tasks

The id of the tasks which have to be completed to fulfill the rule.

points

The amount of points which should be reached.

apiKey

The API key which was generated for the organisation.

Goal

To create a goal all previously created components are needed. So the condition when a goal is completed is defined in the goal rule and the connected tasks. Who can complete a goal is defined by the role for a player and whether it can be done by a group. It is also possible to define whether a goal is repeatable so that the player can complete the tasks and obtains its coins and points as rewards again.

Eg: Creating a Goal

POST

/goal/?name=Goal1&repeatable=true&ruleId=1&rewardIds=1,3,5&roleIds=1&groupGoal=true&apiKey=24bc004f-08ef-413d-988f-b2ca8cc39936

Parameter

Purpose

name

The name of the goal.

repreatable

This parameters states if a goal can be fulfilled more than one time.

ruleId

The rule which has to be fulfilled to reach the goal.

rewardIds

All rewards which can be obtained when all tasks of the goal are fulfilled.

roleIds

Only players with one appropriate role can fulfill the goal and get its rewards.

groupGoal

Defines if a goal can be completed by a group. The default value is false.

apiKey

The API key which was generated for the organisation.

Complete a Task

When a player has completed a Task the following query can be done. The task will be added to the player’s list of finished task. If the task is the last one to fulfill a goal, the goal is also added to the player’s list of finished goals and the player will obtain all its associated rewards.

Eg: Player Sunny (with id 1) completes Task with id 3:

POST

/task/3/complete/1?id=3&playerId=1&finishedDate=2014-02-19 13:21&apiKey=24bc004f-08ef-413d-988f-b2ca8cc39936

Parameter

Purpose

id

The task which is completed.

playerId

The player who has completed the task.

finishedDate

The Date, when the task was completed.

apiKey

The API key which was generated for the organisation.

3. Marketplace

The marketplace gives people the opportunity  to offer tasks that have to be completed to their colleagues so that they are able to fulfil those tasks and obtain the respective reward. Upon creation of a task, an initial bid in terms of coins is set, which will be obtained as additional reward. Via Bids the initial bid can be raised.

Marketplace

To be able to create offers, a marketplace for the organisation is needed. If none exists yet, it first has to be created.

Eg: Creating a marketplace.

deadLine

The date when the task should be done at the latest.

The marketplace where the offer is available.

playerId

4. Presents

Players in a gamification application can send little presents to each other. These presents can be an image or a short text message which contains for example a little praise.  A Board serves a player to send and to store little presents in terms of a short text message or an image.

Presents

A player can send little presents to other players in the form of a text message or a little image message. The difference between these two messages is as the name suggests, that the text message contains a short text and the image message an image.

Eg: Creating a text message and in the second line a message with an image.

POST

Purpose

The player who is the sender of the present.

receiverIds

The player who are the receivers of the present.

content

The content of the message.

imagePath

The URL for the image path so that an image can be sent with the image message.

apiKey

The API key which was generated for the organisation.

5. Donation

When a donation is created, players can pool coins for a certain amount and a certain purpose. If the amount is reached, the goal is fulfilled. Then the purpose can be implemented like a real donation for a charitable purpose or an event for the organisation’s employees.  

Eg: Creating a donation call named “Donation Example” with a goal of 1000 coins.

POST

/donation/?name=Donation Example&description=Reach a certain amount of coins&goal=1000&apiKey=24bc004f-08ef-413d-988f-b2ca8cc39936

Purpose

name

The name of the donation call.

description

The description of the call for donations. Here can be described what happens when the amount is reached.

goal

The amount of coins which should be reached.

apiKey
